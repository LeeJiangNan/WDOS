# WDOS 商场 AI 工单系统架构设计方案 v2.4

> **设计角色**：商场 AI 系统建设架构师
> **设计日期**：2026-06-14
> **版本**：v2.4
>
> **变更说明（v1.0 → v2.0）**：
> - 平台命名由 WOS 更名为 **WDOS**（Work-order Dispatch & Orchestration System）
> - 吸收 WOS Java v2.0.0 的蓝图工作流、多级流程、自定义表单映射等设计
> - 技术栈确定为 **Go 语言**，与原 CRIP 平台保持一致
> - 数据大屏嵌入管理后台（不再独立部署）
> - 报警抑制升级为「抑制 + 锁定」二级机制
>
> **变更说明（v2.0 → v2.1）**：
> - 新增 **排班一键导入**：管理后台支持 Excel 导入排班表，自动关联工单接收人
> - 新增 **多角色看板视图**：领班/主管/经理/总监各自看到超时任务、部门数据、下级统计
> - 新增 **报警图片智能画框**：根据 CRIP callback 中的 `object_list[].rect` 坐标在报警截图上绘制检测框
> - **小程序视频策略**：暂时不展示报警视频（节省带宽和内存），保留视频接口供 Web 端使用
>
> **变更说明（v2.1 → v2.2）**：
> - 多角色看板改为**登录账号驱动**，上级可点击下钻查看下属详细数据
> - 报警图片支持**点击全屏放大** + 双指缩放
> - 工单列表筛选升级为**两级筛选**：算法标签 + 区域分组折叠选择器
> - 新增 **4.10 报警数据统计**：每日每类算法报警数、大区域统计、工单完成率、耗时分布
> - 小程序端**完全去除视频提示条**，后端保留接口
> - SLA 机制改为**倒计时**：超时后仍可接单，仅触发上报
>
> **变更说明（v2.2 → v2.3）**：
> - 确定部署策略：**先同机测试，后独立迁移**（独立数据库 + 配置文件驱动 + Docker 化）
> - 报警数据采用**冗余存储**（图片存入 MinIO、JSON 全量存档），不依赖 CRIP 实时查询
> - 接入方式改为 **Callback 主通道 + 手动补偿**，砍掉 API 自动轮询，降低运维复杂度
>
> **变更说明（v2.3 → v2.4）**：
> - 终端方案改为 **Flutter APP 为主 + 微信小程序兜底**，解决推送实时性问题

## 一、平台定位与命名

### 平台名称

**WDOS** — **W**ork-order **D**ispatch & **O**rchestration **S**ystem（工单调度与编排系统）

### 定位

WDOS 是外置于 CRIP 的独立工单系统，承接 CRIP 报警数据，实现从「AI 报警 → 智能抑制 → 区域路由 → 工单生成 → 多级流转 → 处理闭环 → 统计上报」的全链路管理。

### 与原 WOS 的关系

| 维度 | WOS v1.0 | WOS v2.0.0 (Java) | **WDOS v2.0 (Go)** |
|------|----------|--------------------|---------------------|
| 技术栈 | 未知 | Java + Node-RED 蓝图 | **Go**（与 CRIP 同语言） |
| 工作流 | 固定单线（接单→处理） | Node-RED 蓝图编排（3种流程） | **内置工作流引擎**（可扩展） |
| 表单 | 固定字段 | 自定义组件 + CRIP 字段映射 | **自定义表单 + 字段映射** |
| 抑制 | CRIP 侧间隔去重 | 无增强 | **抑制 + 锁定 + 超时解锁** |
| 路由 | 无 | 无 | **区域路由 + 算法覆盖** |
| 上报 | 无 | 无 | **SLA 逐级上报** |
| 终端 | Web + 鲲云 APP | Web + 鲲云 APP | Web + **Flutter APP + 小程序兜底** |

## 二、CRIP 报警数据接入方案

### 2.1 Callback vs OpenAPI 对比

| 维度 | Callback（CRIP → WDOS） | OpenAPI（WDOS → CRIP） |
|------|--------------------------|--------------------------|
| **数据流向** | CRIP 主动推送 | WDOS 定时拉取 |
| **实时性** | 秒级（报警触发即推送） | 分钟级（取决于轮询频率） |
| **网络要求** | WDOS 需暴露公网 HTTP 端点 | WDOS 能访问 CRIP 即可 |
| **可靠性** | 依赖网络稳定性，WDOS 宕机会丢数据 | WDOS 主动控制，断点续拉 |
| **历史数据** | 仅接收配置之后的报警 | 可回溯查询历史报警 |
| **过滤能力** | CRIP 侧按组合/摄像头/算法配置 | WDOS 侧灵活组合条件查询 |
| **实现复杂度** | 低 | 中（消费位点、token 续期、去重） |
| **资源消耗** | 低（事件驱动） | 中（持续轮询） |

### 2.2 Callback + 手动补偿（推荐）

采用 **Callback 主通道 + 管理后台手动补偿** 的架构。不部署自动轮询 cron 任务。

```
┌────────────────────────────────────────────────────────────────┐
│                        CRIP v3.14.0+                            │
│                                                                │
│  ┌─────────────┐     ┌──────────────────┐                     │
│  │ AI 推理引擎  │────▶│ Callback 推送模块 │                     │
│  │ (视频流分析) │     │ (间隔去重 3-5s)   │                     │
│  └─────────────┘     └────────┬─────────┘                     │
│                               │                                │
│                      OpenAPI 网关 ◀──── 仅在手动补偿时调用      │
└───────────────────────────────┼────────────────────────────────┘
                                │
                                │ 主通道（实时，所有正常流量）
                                ▼
┌────────────────────────────────────────────────────────────────┐
│                         WDOS 工单系统                           │
│                                                                │
│  ┌──────────────────┐                                          │
│  │ Callback 接收器   │                                          │
│  │ POST /api/v1/    │                                          │
│  │ callback/crip    │                                          │
│  └────────┬─────────┘                                          │
│           │                                                    │
│           ▼                                                    │
│  ┌────────────────────────┐                                    │
│  │   报警去重 & 聚合       │                                    │
│  │   snowflake_id 去重    │                                    │
│  └───────────┬────────────┘                                    │
│              ▼                                                 │
│  ┌────────────────────────┐                                    │
│  │   抑制 + 锁定判断       │                                    │
│  └───────────┬────────────┘                                    │
│              ▼                                                 │
│  ┌────────────────────────┐                                    │
│  │   区域路由 + 工单生成   │                                    │
│  └───────────┬────────────┘                                    │
│              │                                                 │
│              │ ═════════ 补偿通道（按需触发）═════════          │
│              ▼                                                 │
│  ┌────────────────────────┐                                    │
│  │   手动补偿器            │  ← 管理员在后台点按钮触发           │
│  │   POST /admin/compensate│  ← 调 CRIP OpenAPI 拉历史报警      │
│  │   → 去重 → 补漏         │  ← 不自动运行，不占 CRIP 资源      │
│  └────────────────────────┘                                    │
└────────────────────────────────────────────────────────────────┘
```

**为什么推荐 Callback + 手动补偿？**

1. **主通道（Callback）**：秒级延迟，处理所有正常报警
2. **手动补偿兜底**：管理员在后台点按钮触发，补漏网络中断/升级停机期间的报警
3. **省运维成本**：不需要 cron 定时器、token 续期、消费位点管理
4. **省 CRIP 资源**：不持续轮询，不给 CRIP OpenAPI 增加负载
5. **商场场景适用**：报警丢几条可以手动补，不像工厂/煤矿零容忍

**什么时候用手动补偿？**
- WDOS 曾停机升级
- 巡检发现某时段工单数异常偏少
- CRIP 通知 Callback 曾中断

**手动补偿技术实现详见 6.5 节。**

### 2.3 接入实施步骤

#### 第一步：CRIP 侧配置

1. **开通 OpenAPI 权限**（仅补偿时需要，日常不依赖）
   - CRIP 管理后台 → 平台管理 → 开放能力 → 接口授权
   - 创建 APP ID 和 Secret

2. **配置 Callback 推送**
   - CRIP → 智能分析 → 报警推送
   - 推送地址：`https://wdos.yourmall.com/api/v1/callback/crip`
   - 推送间隔：3-5 秒（CRIP 侧首次去重）

#### 第二步：WDOS 侧开发

1. **Callback 接收端点**（快速返回 200，写入消息队列）
2. **手动补偿接口**（按需触发，调 CRIP OpenAPI 补漏，详见 6.5 节）
3. **去重聚合层**（`snowflake_id` 为唯一键，Redis SETNX）

### 3.2 WOS v2.0.0 Java 版新增能力

根据《工单系统 WOS 配置说明 v2.0.0》：

#### 3.2.1 蓝图 2.0 工作流引擎

WOS v2.0.0 最大的架构变化是引入了 **Node-RED 蓝图工作流引擎**，作为 CRIP Callback 和 WOS 之间的编排中间层：

```
CRIP Callback ──▶ Node-RED 蓝图 ──▶ WOS API ──▶ 工单创建
                      │
                      ├── callbackFields 字段映射
                      ├── 组合/摄像头/算法 → 工单流程ID 匹配
                      └── 流程路由决策
```

WDOS 将用 **Go 内置工作流引擎** 替代 Node-RED，不依赖外部中间件。

#### 3.2.2 五个内置工作流

| 流程名称 | 描述 | WDOS 继承 |
|----------|------|-----------|
| **工单派单处理** | 报警 → 推送给指定人员 → 指定人员再指派某人处理 | ✅ 保留，扩展为灵活配置 |
| **工单处理审核** | 报警 → 指定人员处理 → 处理完 → 指定上级审核 | ✅ 保留，扩展为多级审核 |
| **一级工单处理** | 低危算法 → 一线人员直接处理 | ✅ 融入 degree 路由 |
| **二级工单处理** | 中危算法 → 一线+班长 | ✅ 融入 degree 路由 |
| **三级工单处理** | 高危算法 → 一线+班长+经理 | ✅ 融入 degree 路由 |

#### 3.2.3 自定义表单 + CRIP 字段映射

WOS v2.0.0 支持在表单中插入自定义组件，并将组件字段 ID 与 CRIP Callback 字段绑定。例如：

| 表单组件 | 字段 ID | 映射 CRIP 字段 | 说明 |
|----------|---------|---------------|------|
| 输入框 | camera_name | `camera_name` | 报警地点 |
| 输入框 | algorithm_name | `algorithm_name` | 报警类型 |
| 输入框 | company | `extra.company` | 扩展字段（公司名称） |
| 图片 | alarm_pic | `alarm_pic_url` | 报警截图 |

WDOS 将保留并增强此能力，支持更多组件类型（下拉框、日期、GPS 定位、签名等）。

#### 3.2.4 Node-RED 工作流关键配置项

从 WOS v2.0.0 的 flow.json 中提取的配置节点：

| 配置项 | 作用 | WDOS 对应 |
|--------|------|-----------|
| `callbackFields` | CRIP 回调字段映射到工单表单 | → 字段映射配置 |
| `wosBaseURL` | WOS API 地址 | → Go 内部调用，不需配置 |
| `组合/摄像头/算法 → 工单ID` | 路由匹配规则 | → 区域路由 + 算法路由 |
| `流程节点连接` | 定义处理→审核流转 | → 工作流引擎 DSL |

## 四、WDOS 增强功能设计

### 4.1 报警抑制与锁定机制

这是 WDOS 相比 WOS 最重要的增强。分两层设计：

#### 第一层：CRIP 侧间隔去重（已有）

CRIP 配置推送时设置间隔（如 3-5 秒），窗口内的重复报警只推送一次。

#### 第二层：WDOS 侧「抑制 + 锁定」（新增）

```
┌─────────────────────────────────────────────────────────────────┐
│                WDOS 报警抑制与锁定引擎                             │
│                                                                 │
│  输入: camera_id + algorithm_id + timestamp + existing_order     │
│                                                                 │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 阶段一：抑制判断（同源同算法去重）                           │  │
│  │                                                           │  │
│  │ Key: {camera_id}:{algorithm_id}                           │  │
│  │                                                           │  │
│  │ if 存在未处理的工单:                                        │  │
│  │   → 【抑制】不生成新工单                                    │  │
│  │   → 在原工单上：                                           │  │
│  │       - duplicate_count += 1                              │  │
│  │       - 更新最新报警图片和视频                              │  │
│  │       - 追加一条操作日志"第N次重复报警"                     │  │
│  │ else:                                                     │  │
│  │   → 生成新工单，写入状态缓存                                │  │
│  └───────────────────────────────────────────────────────────┘  │
│                              │                                    │
│                              ▼                                    │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 阶段二：锁定判断（可选，按策略配置）                          │  │
│  │                                                           │  │
│  │ 当同一摄像头的工单在 X 分钟内未完成时：                       │  │
│  │                                                           │  │
│  │ 锁定模式一：「同算法锁定」                                   │  │
│  │   锁定范围: camera_id + algorithm_id                       │  │
│  │   该摄像头的该算法不再产生报警                               │  │
│  │   其他算法正常报警                                          │  │
│  │                                                           │  │
│  │ 锁定模式二：「全点位锁定」                                   │  │
│  │   锁定范围: camera_id（全部算法）                            │  │
│  │   该摄像头所有算法都不再产生报警                              │  │
│  │   适用于：设备故障、已知施工区域等场景                        │  │
│  │                                                           │  │
│  │ 锁定期间行为:                                               │  │
│  │   → 报警数据记录到 raw_alarm 但不生成工单                    │  │
│  │   → 工单上显示「已抑制 N 条报警」                           │  │
│  │   → 不推送给一线人员（减少疲劳）                             │  │
│  │   → 等待工单超时流程触发                                    │  │
│  └───────────────────────────────────────────────────────────┘  │
│                              │                                    │
│                              ▼                                    │
│  ┌───────────────────────────────────────────────────────────┐  │
│  │ 阶段三：解锁条件                                            │  │
```

### 4.3 人员管理与移动终端

#### 终端策略：Flutter APP 为主 + 微信小程序兜底

微信小程序因推送受限于「订阅消息」机制（用户必须每次手动授权才能发一次），无法保证工单报警的实时触达。因此 WDOS 采用双端策略：

| 终端 | 用户 | 推送 | 安装 |
|------|------|:--:|------|
| **Flutter APP**（主力） | 正式保安、商管员、班长、经理 | ✅ FCM/APNs 秒级推送 | Android APK / iOS IPA |
| **微信小程序**（兜底） | 临时工、外包人员、不便装 APP 的人 | ⚠️ 打开才看到 | 扫码即用 |

**Flutter 选型理由**：

| 维度 | Flutter | 纯 Android | React Native |
|------|:--:|:--:|:--:|
| Android | ✅ | ✅ | ✅ |
| iOS | ✅ | ❌ | ✅ |
| 推送（FCM+APNs） | ✅ 秒到 | ✅ FCM | ✅ |
| 一套代码 | ✅ | ❌ | ✅ |
| 性能 | 高（自绘引擎） | 高 | 中 |

**通知策略**：

| 级别 | 通知方式 | 说明 |
|:--:|------|------|
| 新工单 | APP 推送 + APP 内角标 | FCM/APNs 秒级触达 |
| 超时 L1（30s/150s） | APP 推送 | 班长手机响 |
| 超时 L2（120s/300s） | APP 推送 + 短信 | 双通道保障 |
| 超时 L3（300s/600s） | APP 推送 + 短信 + 电话 | 最高优先级 |

**小程序保留用途**：
- 临时工/外包人员查看工单（不需要实时推送）
- 管理人员不方便装 APP 时快速查看
- 扫码即可分享工单给其他人

API 统一，两个终端共用同一套 65 个接口，仅登录方式不同（APP 用手机号+密码，小程序用微信授权）。

```
  │              状态: 待接单 → 处理中
  │              处理 SLA 计时开始
  │

  │              审核 SLA 计时开始
```

### 4.5 工作流引擎

替代 WOS v2.0.0 的 Node-RED 蓝图，WDOS 内置 **Go 工作流引擎**。

#### 设计理念

- **状态机驱动**：每个工单是一个有限状态机实例

### 4.8 报警图片智能画框

#### 业务背景

CRIP Callback 中的报警图片是原始截图，不包含检测框标注。WDOS 需要在展示报警图片时，根据 callback 中的检测坐标信息，在图片上绘制检测框，让处理人一眼看到「哪里触发了报警」。

#### 数据来源

CRIP Callback 中的检测目标坐标：

```json
"result_data": [{
  "result_data": {
    "task_result": {
      "object_list": [
        {
          "class_id": 0,
          "rect": {
            "x": 224,      // 检测框左上角 X 坐标
            "y": 605,      // 检测框左上角 Y 坐标
            "width": 206,  // 框宽度
            "height": 349  // 框高度

          "score": 0.853873,  // 置信度
          "extra_data": "[{\"cn_name\":\"人头检测分数\",\"name\":\"head_det_score\",\"vals\":[0.93]}]"
        }

}]
```

#### Go 代码示例（画框核心逻辑）

```go
package alarm

import (
    "image"
    "image/color"
    "image/draw"
    "image/jpeg"
    "net/http"
    "github.com/fogleman/gg"
)

type DetectionBox struct {
    X      int     `json:"x"`
    Y      int     `json:"y"`
    Width  int     `json:"width"`
    Height int     `json:"height"`
    Score  float64 `json:"score"`
    Label  string  `json:"label"`
}

func AnnotateImage(imgURL string, boxes []DetectionBox) ([]byte, error) {
    // 1. 下载原始图片
    resp, _ := http.Get(imgURL)
    defer resp.Body.Close()
    srcImg, _, _ := image.Decode(resp.Body)

    // 2. 创建绘制上下文
    bounds := srcImg.Bounds()
    dc := gg.NewContext(bounds.Dx(), bounds.Dy())
    dc.DrawImage(srcImg, 0, 0)

    // 3. 逐个绘制检测框
    for _, box := range boxes {
        // 按置信度选择颜色：>0.8 红 / >0.6 橙 / 其他 绿
        boxColor := color.RGBA{255, 0, 0, 255}
        if box.Score < 0.8 { boxColor = color.RGBA{255, 150, 50, 255} }
        if box.Score < 0.6 { boxColor = color.RGBA{0, 200, 80, 255} }

        dc.SetColor(boxColor)
        dc.SetLineWidth(2)
        dc.DrawRectangle(float64(box.X), float64(box.Y),
            float64(box.Width), float64(box.Height))
        dc.Stroke()

        // 绘制标签
        label := fmt.Sprintf("%s %.0f%%", box.Label, box.Score*100)
        dc.SetRGB(1, 1, 1)
        dc.DrawString(label, float64(box.X), float64(box.Y)-5)
    }

    // 4. 编码输出
    var buf bytes.Buffer
    jpeg.Encode(&buf, dc.Image(), &jpeg.Options{Quality: 90})
    return buf.Bytes(), nil
}
```

### 4.9 小程序视频策略

#### 决策

小程序端**不展示报警视频**，视频播放仅在 Web 管理后台提供。前端不渲染视频组件，不加载视频 URL，不显示视频提示条。后端 API 完整保留 `video_url` 字段，不做任何裁剪。

| 考量 | 说明 |
|------|------|
| **带宽成本** | 单个报警视频 5-15MB，商场日均 2000+ 条报警，小程序端播放会消耗大量移动流量 |
| **内存占用** | 小程序 WebView 内存有限，播放视频容易导致卡顿甚至崩溃 |
| **加载速度** | 一线人员需要在 30 秒内快速接单，视频加载延迟影响响应速度 |
| **实际需求** | 一线人员更依赖报警截图（含检测框）判断情况，视频回放更多是事后复盘用 |

#### 各端策略

| 端 | 报警图片 | 报警视频 | 视频提示条 | 说明 |
|------|:--:|:--:|:--:|------|
| **微信小程序** | ✅ 标注图（可放大） | ❌ 不展示 | ❌ 不显示 | 前端不渲染视频组件，不加载 video_url |
| **Web 管理后台** | ✅ 标注图 | ✅ 可播放 | — | 复盘、审核场景使用 |
| **API 接口** | ✅ `alarm_pic_url` | ✅ `video_url` | — | 完整保留，不做删减 |

### 4.11 SLA 倒计时机制

#### 行为变更

接单倒计时结束后**仍然可以接单**，但会触发上报流程：

```
倒计时 T=30秒 → 倒计时归零
  │
  ├── 工单不锁定，仍然可接单
  ├── 系统自动上报班长（一级上报）
  └── 倒计时显示变为 "已超时 XX 秒"

倒计时 T=120秒 → 二级上报经理，仍可接单
倒计时 T=300秒 → 三级上报总监，仍可接单
```

| 时间点 | 工单状态 | 操作 | 上报 |
|--------|----------|------|------|
| 0-30秒 | 正常待接单 | 可接单 | 无 |
| 30-300秒 | 超时待接单 | 仍可接单 | 逐级上报 |
| >300秒 | 严重超时 | 仍可接单 | 总监已收到 |
| 有人接单 | 处理中 | 进入处理流程 | 上报停止 |
| 处理完成 | 已完成 | — | 解除锁定 |

| 端 | 报警图片 | 报警视频 | 说明 |
|------|:--:|:--:|------|
| **微信小程序** | ✅ 展示（含检测框） | ❌ 不展示 | 仅展示标注后的截图 |
| **Web 管理后台** | ✅ 展示（含检测框） | ✅ 可播放 | 复盘、审核场景使用 |
| **API 接口** | ✅ `alarm_pic_url` | ✅ `video_url` | 完整保留，不做删减 |

#### API 接口保留

视频接口不做任何裁剪，CRIP Callback 中的 `video_url` 字段正常接收并存储：

```json
// 工单详情 API 响应中 video_url 字段正常返回
{
  "video_url": "https://wdos.yourmall.com/files/minio/video_1525.mp4",
  "video_available": false  // 小程序端标识，前端据此隐藏播放器
}
```

后续如果需要在小程序端启用视频播放，只需：
1. 前端移除 `video_available` 判断
2. 开启视频组件渲染
3. 无需后端改动

### 6.4 数据独立与冗余存储策略

#### 核心原则：WDOS 不依赖 CRIP 做实时查询

WOS 1.0 的教训：工单详情页实时调 CRIP API 获取报警数据，CRIP 删数据 → WOS 报错。

WDOS 的对策：**收到 Callback 时立即将所有报警数据冗余存储到自己的数据库和文件系统。**

```
CRIP Callback 推送
      │
      ▼
┌──────────────────────────────────────────────────┐
│ 1. raw_json 全量存储（MySQL JSON 列）              │
│    → 完整保留 Callback 全部字段，永不过期          │
│    → 即使以后 CRIP 加了新字段，raw_json 里也有     │
├──────────────────────────────────────────────────┤
│ 2. 关键字段提取到索引列                            │
│    snowflake_id, camera_id, algorithm_id, degree │
│    → 方便查询筛选，不必每次解析 JSON              │
├──────────────────────────────────────────────────┤
│ 3. 报警图片下载到 MinIO                           │
│    - 原始图: /alarms/raw/{snowflake_id}.jpg       │
│    - 标注图: /alarms/anno/{snowflake_id}.jpg      │
│    → 工单引用自己的 MinIO 地址，不调 CRIP         │
├──────────────────────────────────────────────────┤
│ 4. 视频 URL 保留原值（不下载）                     │
│    → 视频太大（5-15MB），仅保留 CRIP 原始 URL     │
│    → 视频不可播放不影响工单其他功能                │
│    → Web 管理后台直接引用 URL 播放                │
└──────────────────────────────────────────────────┘
      │
      ▼
  工单详情永远查自己的 MySQL + MinIO
  CRIP 删数据 → WDOS 不受影响 ✅
```

#### 对比 WOS 1.0

| 维度 | WOS 1.0 | WDOS |
|------|---------|------|
| 报警数据存储 | CRIP 数据库（仅存引用 ID） | WDOS MySQL（raw_json + 索引列） |
| 报警图片 | CRIP 文件服务器（实时引用） | WDOS MinIO（下载复制） |
| 报警视频 | CRIP（实时引用） | CRIP URL（保留，不下载） |
| 打开工单详情 | 实时调 CRIP API | 查自己的 MySQL |
| CRIP 删数据 | 工单报错 ❌ | 工单正常 ✅ |
| CRIP 不可用 | 历史工单全部不可用 ❌ | 历史工单正常 ✅ |

### 6.5 手动数据补偿

替代 API 自动轮询，采用管理员手动触发的一次性补偿机制。

#### 使用场景

- CRIP Callback 曾因网络故障中断
- WDOS 曾停机升级，期间报警遗漏
- 管理员巡检发现某时段工单数量异常偏少

#### 技术实现

```
管理员点「开始补偿」
      │
      ▼
WDOS → POST /mc/v1/authenticate → CRIP 获取 token
      │
      ▼
WDOS → POST /cb/v1/log/search → CRIP 分页拉取报警日志
      │
      ▼
逐条去重（Redis snowflake_id）
  ├── 已存在 → 跳过
  └── 不存在 → 写入 raw_alarm → 走正常工单生成流程
      │
      ▼
返回结果给管理员:
  {
    "total_fetched": 2156,
    "newly_added": 3,
    "skipped": 2153,
    "duration": "12.3s"
  }
```

#### 补偿接口

```
POST /api/v1/admin/compensate
Authorization: Bearer <token>

请求体:
{
  "start_time": "2026-06-14 08:00:00",
  "end_time":   "2026-06-14 12:00:00"
}

响应:
{
  "code": 0,
  "data": {
    "total_fetched": 2156,
    "newly_added": 3,
    "skipped": 2153,
    "duration": "12.3s"
  }
}
```

#### 与旧方案的对比

| 维度 | 旧：API 自动轮询 | 新：手动补偿 |
|------|:--:|:--:|
| 触发方式 | cron 每 30 秒 | 管理员手动点击 |
| 开发量 | cron + token 续期 + 消费位点 | 一个接口 |
| 运维负担 | 持续占用 CRIP 资源 | 按需触发 |
| 数据保障 | 自动兜底 | 人工发现后补充 |
| 适合场景 | 零容忍丢数据 | 商场场景，允许少量遗漏后手动补 |
