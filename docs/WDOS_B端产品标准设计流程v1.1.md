## 三、P0/P1/P2 开发优先级

### 分级标准

| 级别 | 含义 | 判断标准 |
|:--:|------|----------|
| **P0** | 必须做 | 没有它，核心业务流程跑不通 |
| **P1** | 应该做 | 没有它系统能用但体验很差 |
| **P2** | 可以后面做 | 锦上添花，不影响主流程 |

### 阶段 4：开发测试

```
开发阶段:
☐ 技术方案评审（后端架构 + 数据库设计 + 缓存策略）
☐ 接口 Mock（后端没写完时前端用 Mock 数据开发）
☐ 每日站会（昨天做了什么、今天做什么、有什么阻塞）
☐ Code Review（每个 PR 至少一人 Review）

测试阶段:
☐ 单元测试（Go test coverage > 70%）
☐ 接口测试（所有 API 至少一条正常 + 三条异常用例）
☐ 端到端测试（完整链路: CRIP → Callback → 工单 → 小程序接单 → 处理 → 完成）
☐ 异常测试:
    - 工单系统宕机 → 重启后 Redis Stream 消费是否恢复
    - CRIP Callback 停止推送 → API 同步器是否自动接管
    - Redis 满 → 报警数据是否丢失
    - 微信小程序弱网 → 接单操作是否超时重试
☐ 压力测试（模拟日均 2000+ 报警场景）

UAT（用户验收测试）:
☐ 准备 UAT 环境（独立部署，不影响生产）
☐ 培训材料（操作手册 or 录屏教程）
☐ 真实用户试用（至少安保部 2-3 名一线人员 + 1 名班长 + 1 名经理）
☐ 收集反馈 → 评估是否阻塞发版
```

### 阶段 5：上线运维

```
上线前:
☐ 数据库备份
☐ 回滚方案（K8s rollback + DB migration down）
☐ 监控告警配置（CPU/内存/Redis/MySQL/接口错误率）
☐ 上线公告（通知所有用户）

灰度发布:
☐ 先切 10% 流量（或先让安保部使用）
☐ 观察 48 小时（错误率、响应时间、用户反馈）
☐ 无问题 → 全量

全量上线:
☐ 全量切换
☐ 7×24 小时监控值班（第一周）
☐ 每日巡检（错误日志、慢查询、存储空间）

持续迭代:
☐ 收集用户反馈（小程序内置反馈入口）
☐ 每两周一次迭代规划
☐ 每月一次版本回顾
```

## 六、变更控制流程

当需求发生变更时，按变更类型走不同流程：

| 变更类型 | 审批 | 示例 |
|----------|------|------|
| **PATCH**（修 bug/文案） | 开发自决 | 按钮颜色不对、倒计时数字偏了 |
| **MINOR**（加功能/改参数） | 产品确认 | 新增导出格式、SLA 150s→180s |
| **MAJOR**（改流程/删功能） | 产品 + 技术负责人 | 工单增加确认环节、删除旧流程 |

```
变更提出 → 评估影响面 → 确定优先级 → 更新 PRD → 开发 → 测试 → 上线

  关键原则:
  1. 任何 MAJOR 变更必须更新设计文档 + 原型
  2. 原型是唯一的 UI 真相来源（不要存在「开发自己觉得好看就改了」）
  3. 接口变更必须同步更新 API 文档
  4. 权限变更必须同步更新权限矩阵表
```
