# WDOS 关键决策记录

> **日期**：2026-06-21
> **用途**：记录设计过程中所有「为什么选 A 不选 B」的关键决策，防止以后换人接手不知道为什么这样设计。

---

## 一、架构决策

### D1. 平台命名：WOS → WDOS

| 选项 | 说明 |
|------|------|
| A: 沿用 WOS | 和原系统同名，容易混淆版本 |
| **B: WDOS** | ✅ **Work-order Dispatch & Orchestration System**，强调「调度+编排」的定位 |

### D2. 部署策略：独立部署 vs 内嵌 CRIP

| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 内嵌 CRIP（WOS 1.0 方式） | 通讯零延迟、部署简单 | CRIP 升级 = WDOS 升级、宕机一起挂、数据库耦合 |
| **B: 独立部署（先同机后迁移）** | ✅ 代码独立、升级互不影响、可随时迁移 | 需要独立维护基础设施 |

**决策**：B。从第一行代码起就为独立迁移做准备——独立 database、配置文件驱动、Docker 化。先在 CRIP 服务器上测试验证，后面视规模决定是否独立迁移。

参考：`WDOS商场AI工单系统架构设计方案v2.0.md` 第 6.3 节

---

### D3. 数据接入：Callback vs Callback + API 轮询

| 选项 | 说明 |
|------|------|
| A: Callback + API 自动轮询 | 双通道兜底，cron 每 30 秒拉一次 |
| **B: Callback + 手动补偿** | ✅ 只保留 Callback，API 补偿改为管理员手动触发 |

**决策**：B。砍掉 cron 定时器、token 续期、消费位点管理。对于商场场景（非煤矿/工厂），报警丢几条可以手动补回来，不需要自动轮询的运维负担。

技术细节：
- 日常靠 Callback 推送（秒级延迟）
- 管理员在后台点「数据补偿」→ 调 CRIP OpenAPI 拉取指定时间段 → snowflake_id 去重补漏
- 心跳检测：5 分钟无 Callback → 管理后台标黄告警

参考：`WDOS_业务流程闭环图.md` Q3

---

### D4. 数据存储：冗余存储 vs 实时引用

| 选项 | 说明 |
|------|------|
| A: 实时引用 CRIP（WOS 1.0 方式） | 只存 CRIP 数据 ID，打开工单详情时实时调 CRIP API |
| **B: 冗余存储** | ✅ 收到 Callback 时立即把数据全量存到 WDOS 自己的 MySQL + MinIO |

**决策**：B。WOS 1.0 的教训——CRIP 删了报警数据，WOS 工单就打不开了。WDOS 方案：
- `raw_json` 全量存入 MySQL（永远不丢）
- 报警图片下载到 MinIO（不再依赖 CRIP 文件服务器）
- 视频 URL 保留原值不下载（太大）
- 工单详情永远查自己的数据库

参考：`WDOS商场AI工单系统架构设计方案v2.0.md` 第 6.4 节

---

### D5. 实时性保障：心跳检测 vs 自动 API 同步

| 选项 | 说明 |
|------|------|
| **B: 心跳检测** | ✅ Redis 计数器，每次 Callback +1，cron 每 5 分钟检查一次。归零 = 告警 |

**决策**：B。配合手动补偿，管理员看到告警后自行判断是否需要补数据。不自动拉取（避免 CRIP 刚恢复就涌来几千条积压报警）。

参考：`WDOS_业务流程闭环图.md` Q3

## 二、技术选型

### D6. 后端语言：Go 1.22

| 选项 | 说明 |
|------|------|
| A: Java（WOS 2.0 用 Java） | 鲲云有 Java 版 WOS 可供参考 |
| **B: Go** | ✅ 与 CRIP 平台同语言（Go），方便联调维护，团队其他 Go 开发者也能接手 |

### D7. 移动端：微信小程序 vs 独立 APP

| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 独立 APP（鲲云 AI 助手） | 推送及时 | 需要下载安装、iOS & Android 双端开发 |
| B: 微信小程序 | 扫码即用、零安装 | 微信订阅消息不可靠，无法实时触达 |
| **C: Flutter APP（主力）+ 微信小程序（兜底）** | ✅ 推送实时（FCM/APNs）+ 兜底覆盖 | 需要额外维护一个小程序 |

**决策**：C（v2.4 更新）。微信订阅消息机制限制太大（用户必须每次手动授权才能发一次），无法保证工单报警的实时触达。Flutter 一套代码出 Android APK + iOS IPA，FCM/APNs 原生推送秒级触达。小程序保留给临时工/外包人员，不依赖推送，打开就能看。

### D9. 编译参数：CGO_ENABLED=0

macOS M5 编译 Go 二进制时 dyld 报 `missing LC_UUID`。

**解决**：`CGO_ENABLED=0 go build`，静态链接，无 C 依赖。

写入 Makefile 和 `.trellis/spec/config-and-infra.md`。

## 三、业务逻辑决策

### D10. 报警抑制策略：按次数 vs 按时长

| 选项 | 说明 |
|------|------|
| A: 累计 N 次后强制生成新工单 | 2 分钟 20 次报警 → 生成 |
| **B: 按时长抑制** | ✅ 同摄像头+同算法有未处理工单 → 一直抑制 |

**决策**：B。部分算法（如烟雾检测）容易误报，按时长即可，避免频繁重复工单。

参考：`WDOS_业务流程闭环图.md` Q9

---

### D11. SLA 机制：超时锁定 vs 倒计时不锁定

| 选项 | 说明 |
|------|------|
| A: 超时后锁定工单 | 倒计时归零 → 工单不可接 |
| **B: 超时后仍可接单，仅触发上报** | ✅ 倒计时归零 → 上报班长/经理/总监，但工单仍然可以接和处理 |

**决策**：B。商场场景不同于工厂，超时的工单可能只是保安在忙没看到，不能因为超时就拒绝他接单。

上报时间线：
- 接单：30s(L1班长) → 120s(L2经理+锁定) → 300s(L3总监)
- 处理：150s(L1) → 300s(L2) → 600s(L3)

参考：`WDOS商场AI工单系统架构设计方案v2.0.md` 第 4.4 节

---

### D12. 同部门双领班超时通知

| 选项 | 说明 |
|------|------|
| A: 按排班时间匹配当班班长 | 需要查排班表 |
| **B: 两个都通知** | ✅ 白班和夜班班长都收到，谁先处理算谁的 |

**决策**：B。后续 SLA 策略加配置项，允许基于业务修改通知规则。

参考：`WDOS_业务流程闭环图.md` Q7

---

### D13. 工单没人管怎么办

| 选项 | 说明 |
|------|------|
| A: 自动转给管理员 | 管理员变成兜底人 |
| **B: 一直标红** | ✅ 不自动转交，持续高亮提醒 |

**决策**：B。自动转交会让管理员成为「垃圾桶」，所有没人管的工单都堆过去。一直标红至少让人知道有问题。

参考：`WDOS_业务流程闭环图.md` Q1

---

### D14. 线下处理：要不要手动建工单入口

| 选项 | 说明 |
|------|------|
| A: 管理后台加「手动创建工单」 | 保安口头处理完的报警可以补记录 |
| **B: 暂时不管** | ✅ 先不管线下场景 |

**决策**：B。第一个版本专注于线上报警闭环。

参考：`WDOS_业务流程闭环图.md` Q5

---

### D15. 小程序不展示视频

| 选项 | 说明 |
|------|------|
| A: 展示视频播放器 | 完整功能 |
| **B: 不展示** | ✅ 节省移动带宽和内存 |

**决策**：B。单个视频 5-15MB，一线人员需要在 30 秒内快速判断情况，视频加载延迟影响响应速度。截图的标注框已经足够。视频回放仅在 Web 管理后台提供。API 接口完整保留 `video_url`，不做裁剪。

参考：`WDOS商场AI工单系统架构设计方案v2.0.md` 第 4.9 节

---

### D16. 报警图片画框

利用 CRIP Callback 中的 `object_list[].rect` 坐标，在报警截图上绘制检测框。

- 框颜色按置信度：>80% 红 / 60-80% 橙 / <60% 绿
- 图片标注在 Go 后端处理（`image/draw` 或 `gg` 库）
- 标注后图片单独存储到 MinIO `/alarms/anno/{snowflake_id}.jpg`
- 前端支持切换「标注图 / 原图」

参考：`WDOS商场AI工单系统架构设计方案v2.0.md` 第 4.8 节

## 五、接口设计决策

### D19. 统计接口角色联动

同一个接口 URL，不同角色调用时返回不同范围的数据。后端从 JWT token 读取角色，自动加 WHERE 条件限定数据范围。

```
handler    → 个人数据
supervisor → 所在组数据
manager    → 所在部门数据
director   → 全商场数据
```

参考：`WDOS_接口对账表.md` A2

---

### D20. API 版本管理

破坏性变更走 `/api/v2/`，非破坏性在 v1 里加字段。

**原因**：微信小程序有 1-7 天审核期。如果直接改 v1 接口，审核期内的旧版小程序会全部报错。用 v2 可以并行运行两个版本，旧版不受影响。

参考：`WDOS_接口对账表.md` A6

---

### D21. 文件上传方式

直接在 `submit`（工单提交）和 `import`（排班导入）接口接收 multipart/form-data，不建通用上传接口。

**原因**：工单现场照片和排班 Excel 都是一次性的，不需要复用。一次请求搞定，简单直观。

参考：`WDOS_接口对账表.md` A3

## 六、基础设施决策

### D22. Docker 端口映射

| 服务 | 容器端口 | 宿主机端口 | 原因 |
|------|:--:|:--:|------|
| MySQL | 3306 | 3307 | 避免和本地 MySQL 冲突 |
| Redis | 6379 | 6380 | 同上 |
| MinIO | 9000 | 9000 | 和默认一致 |
| MinIO Console | 9001 | 9001 | 和默认一致 |

### D23. Redis Key 前缀隔离

所有 key 统一加 `wdos:` 前缀（从 `config.yaml` 的 `redis.prefix` 读取），确保和其他系统共用 Redis 时不会冲突。

## 七、Go 代码规范决策

### D24. GORM JSON 列处理

MySQL JSON 列不接受空字符串，必须用 `*string`（指针类型，空值写 NULL）或初始化默认值 `"{}"`。

踩坑记录：2026-06-21，`work_order.form_data` 空字符串导致 `Error 3140: Invalid JSON text`。

参考：`.trellis/spec/models.md`
